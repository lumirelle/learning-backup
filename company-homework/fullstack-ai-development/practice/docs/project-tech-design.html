2. 数据库 Schema 说明

完整 DDL 见 docs/project-schema.sql，此处列出核心表设计要点。

表名行数量级核心职责关键约束
users 百级 存储所有角色的账号信息 email UNIQUE；密码必须 BCrypt(cost≥12)
departments 十级 部门字典表 name UNIQUE
jobs 百级 岗位信息，状态机 DRAFT→OPEN→CLOSED GIN 全文索引支持关键词搜索
applications 千~万级 候选人申请记录，核心状态机（8 阶段） UNIQUE(job_id, candidate_id) 防重投
stage_logs 万级 阶段流转审计日志，只增不删 无 UPDATE 权限，确保不可篡改
interview_records 千级 面试评价，可多轮存储 rating BETWEEN 1 AND 5
refresh_tokens 百级 JWT RefreshToken 持久化 存 SHA-256(token)；定期清理过期记录

2.1 枚举类型

类型名值说明
user_roleADMIN · HR · CANDIDATE控制接口权限和数据可见范围
job_statusDRAFT → OPEN → CLOSED岗位状态机，只能向前推进
application_stageAPPLIED · SCREENING_PASS · PHONE_INTERVIEW · TECH_INTERVIEW · HR_INTERVIEW · OFFER · HIRED · REJECTED候选人流转阶段，HIRED/REJECTED 为终态
interview_conclusionPASS · REJECT · HOLD面试轮次评价结论

2.2 索引策略

  • users.email：B-tree，登录查询高频
  • jobs.status：B-tree，候选人列表仅查 OPEN 状态
  • jobs 全文：GIN（title + location），支持关键词模糊搜索
  • applications(job_id, candidate_id)：复合唯一索引，防重投同时加速查询
  • applications.stage：B-tree，看板按阶段分组聚合
  • stage_logs.application_id：B-tree，加速候选人详情时间线查询
  • refresh_tokens.expires_at：B-tree，定时清理过期 Token

3.1 各层职责说明

Controller 层
  • 接收 HTTP 请求，绑定路径/查询参数
  • 使用 @Valid 触发 DTO 参数校验
  • 不含业务逻辑，仅做数据转换和调用 Service
  • 返回统一 ApiResponse<T> 包装
Service 层
  • 所有业务规则在此层实现
  • @Transactional 管理跨表事务
  • 状态机校验：非法流转抛 BizException
  • 写 stage_log 与更新 application 同一事务
Repository 层
  • MyBatis-Plus BaseMapper 提供通用 CRUD
  • 复杂多表查询写 XML Mapper
  • 不含业务逻辑，只做数据读写
  • 分页统一使用 Page<T> 对象
Model 层
  • Entity：与数据库表字段一一对应
  • DTO：入参，含 JSR-303 注解
  • VO：出参，脱敏、聚合后返回
  • Entity/VO 转换用 MapStruct（推荐）
Exception 层
  • GlobalExceptionHandler 捕获所有异常
  • 区分 4xx（业务错误）和 5xx（系统错误）
  • 统一返回 { code, msg, data: null }
  • 生产环境屏蔽异常堆栈信息
Config / Middleware
  • JwtAuthFilter：每请求从 Header 取 Token 并验签
  • SecurityConfig：放行 /auth/** 和 GET /jobs
  • 角色权限用 @PreAuthorize("hasRole('HR')")
  • CORS 配置允许前端开发端口

3.2 后端编码规范

接口响应格式

  • 成功：{ "code": 200, "msg": "ok", "data": {...} }
  • 业务失败：{ "code": 40001, "msg": "岗位不存在", "data": null }
  • HTTP 状态码与 code 字段保持一致
  • 分页响应：data: { list, total, page, pageSize }

错误码规范

  • 10001：Token 无效/过期
  • 10002：权限不足
  • 20001：资源不存在
  • 20002：状态流转非法
  • 20003：重复投递
  • 30001：文件类型不合法

状态机约束（ApplicationService）

  • 合法流转图由 Map<Stage, Set<Stage>> 声明
  • HIRED / REJECTED 为终态，不可再次变更
  • 流转同时写 stage_logs（同一事务）
  • 拒绝操作必须携带 reject_reason

安全规范

  • 密码 BCrypt cost=12，禁止日志输出
  • JWT AccessToken 有效期 2h，RS256 签名
  • RefreshToken SHA-256 后存库
  • 文件接口鉴权：简历 PDF 仅 HR 可访问
  • 防止路径穿越：文件名 UUID 化存储

4.1 模块职责说明

views/ 页面层
  • 每个文件对应一个路由，<script setup> 风格
  • 页面级请求在 onMounted / watchEffect 触发
  • 不含复杂 UI 逻辑，拆分为子 components
  • HR 页面放 views/hr/ 目录统一管理
components/ 组件层
  • 仅接收 props，通过 emit 向上通知
  • 看板拖拽用 vuedraggable 包裹 KanbanCard
  • 图表组件封装 ECharts，props 传数据
  • PDF 预览用 <iframe> 或 pdfjs-dist
stores/ 状态层
  • useAuthStore：持久化 Token 到 localStorage
  • 提供 isLoggedIn、userRole computed
  • Token 刷新逻辑在 request.ts 拦截器中触发
  • 登出时清除所有本地状态
api/ 请求层
  • request.ts：统一注入 Authorization Header
  • 响应拦截：401 → 自动刷新 Token，再重试
  • 每个模块文件导出具名函数，类型安全
  • 文件上传单独处理 multipart/form-data
router/ 路由层
  • meta.roles 声明页面允许的角色
  • 全局守卫检查 Token + 角色，无权则 redirect
  • History 模式，Nginx 需配置 try_files
  • 懒加载：所有页面 () => import(...)
utils/ + types/ 工具层
  • stageMap.ts：枚举 → 中文 / Naive UI tag 颜色
  • format.ts：dayjs 格式化时间
  • api.d.ts：与后端 VO 严格对齐
  • 枚举常量统一从 enums.ts import

4.2 前端编码规范

组件规范

  • 全部使用 <script setup lang="ts">
  • Props 用 defineProps<{ ... }>() 类型声明
  • Emit 用 defineEmits<{ ... }>()
  • 组件文件名 PascalCase，路由文件 XxxView.vue

状态管理规范

  • Store 以 use 前缀命名（Composition API 风格）
  • 异步操作统一放 action，不在组件内直接 await api
  • 需要持久化的字段用 pinia-plugin-persistedstate

加载 & 错误处理

  • 所有异步操作展示 NButton :loading 或 NSpin
  • 请求失败用 useMessage().error(msg) 提示
  • 表单提交前 formRef.value?.validate()
  • 空态 / 骨架屏：列表为空时展示引导文案

代码质量

  • ESLint + OXLint 统一格式化（参考 eslint.config.js）
  • 禁止 any，后端类型从 api.d.ts import
  • 图标统一用 Naive UI 内置图标或 xicons
  • 颜色统一用 Naive UI 主题 token，不硬编码色值

5. Docker Compose 部署方案

以下为 docker-compose.yml 服务规划，一条命令启动全部服务：

服务名镜像端口职责依赖
postgres postgres:16-alpine 5432 主数据库，挂载 volume 持久化 —
redis redis:7-alpine 6379 Token 缓存，无需持久化 —
backend 自建（Spring Boot fat jar） 8080 后端 API 服务 postgres, redis
frontend 自建（Vite build + nginx） 80 / 443 静态资源 + 反代 /api → backend:8080 backend
本地启动命令

docker compose up -d — 所有服务后台启动
docker compose logs -f backend — 查看后端日志
docker compose down -v — 停止并清除 volume（慎用）

6. 开发启动检查清单

后端启动前检查

  • ✅ PostgreSQL 已启动，执行 project-schema.sql 初始化表结构
  • ✅ Redis 已启动（redis-cli ping 返回 PONG）
  • ✅ .env 文件已复制自 .env.example 并填写真实值
  • ✅ JWT 密钥对已生成（openssl genrsa + openssl rsa -pubout）
  • ✅ 上传目录 UPLOAD_PATH 存在且有写权限

前端启动前检查

  • ✅ Node.js ≥ 20，pnpm install 完成
  • ✅ .env.local 中 VITE_API_BASE_URL 指向后端地址
  • ✅ vite.config.ts proxy 已配置 /api → http://localhost:8080
  • ✅ pnpm run lint 无报错
重要提示

初始管理员密码在 project-schema.sql 种子数据中为占位符， 部署前必须替换为真实 BCrypt hash（可用 htpasswd -bnBC 12 "" your_password | tr -d ':\n' 生成）。
