ATS 招聘管理系统 — Phase 3 开发计划

版本：v1.0 状态：待评审 日期：2026-05-21 关联：project-requirements.html · project-schema.sql · project-tech-design.html

1. 目标与产出

阶段定位

Phase 3 是编码实现阶段：把 Phase 2 的 Schema 与技术设计落地为可运行的前后端代码， 产出可演示的 MVP，并为 Phase 4 的部署交付做好准备。本计划以 10 个工作日 为基准节奏， 可根据实际投入按比例伸缩。

输入（来自 Phase 1 / 2）

  • 需求文档：docs/project-requirements.html
  • 数据库 DDL：docs/project-schema.sql
  • 技术设计：docs/project-tech-design.html
  • 已确定的技术栈：Spring Boot 3 · Vue 3 · PostgreSQL 16 · Redis 7

输出（Phase 3 末态）

  • 后端仓库 ats-backend/，本地可运行
  • 前端仓库 ats-frontend/，本地可运行
  • 开发用 docker-compose.yml（含 postgres / redis）
  • 初始化脚本 / 演示数据 / API 自测脚本
  • 每个里程碑结束的可演示版本

2.1 核心原则

  • 纵切而非横切：每个里程碑都打通「后端 → 前端 → 联调」一条完整链路，避免后端先写一个月再前端联调时大返工
  • 主线优先：候选人投递 → HR 看板流转 是业务主线，优先于辅助模块
  • 契约先行：每个接口先与 project-tech-design.html 的接口约定对齐，再让 AI 生成代码
  • 小步快跑：单次 AI 生成代码量 ≤ 1 个 Controller + Service + Mapper，生成后立即跑通再继续
  • 状态机闭环：每次写入 applications.stage 必须同时写 stage_logs，在同一事务

4.2 M1 · 认证模块（Day 2 – Day 3）

后端：认证
M1-BE
实现要点
  • 实体：User + UserMapper（MyBatis-Plus）
  • DTO：RegisterRequest（含 @Email、@Size(min=8)）/ LoginRequest / RefreshRequest
  • VO：UserVO（脱敏 password_hash）/ TokenPair
  • JwtConfig：RS256 加载密钥、签发/解析 Access & Refresh
  • JwtAuthFilter：从 Header 提取 Token，注入 SecurityContext
  • SecurityConfig：白名单 /auth/**、GET /jobs/**；其他需鉴权
  • RefreshToken 哈希入 Redis（key=rt:{userId}:{tokenHash}，TTL=7d）
  • GlobalExceptionHandler + BizException + ErrorCode 落地
接口
  • POST /api/v1/auth/register
  • POST /api/v1/auth/login
  • POST /api/v1/auth/refresh
  • POST /api/v1/auth/logout
  • GET /api/v1/auth/me
前端：登录态
M1-FE
实现要点
  • LoginView.vue / RegisterView.vue：Naive UI NForm + 校验
  • useAuthStore：user、accessToken、refreshToken，持久化到 localStorage
  • api/auth.ts：register / login / refresh / logout / me
  • request.ts：请求拦截注入 Authorization，响应拦截 401 → 自动 refresh → 重试原请求（队列防重入）
  • router/index.ts：全局 beforeEach 守卫（未登录跳 /login、角色不符跳 403）
UX 细节
  • 登录按钮 loading 状态；错误 toast；回车提交
  • 已登录用户访问 /login 直接跳首页
联调验收
M1-INT
必跑场景
  • 注册一个 HR 账号 → 登录 → 拿到 token → 访问 /auth/me 成功
  • 故意把 access token 改坏 → 401 → 拦截器走 refresh → 重试成功
  • logout → access/refresh 均被吊销，/me 返回 401
  • 未登录访问 HR 页面 → 自动跳 /login
注意事项
⚠ Risk
  • 密码 BCrypt cost=12，禁止任何日志输出 raw password
  • refresh 失败要清空本地 token，避免死循环 401 → refresh → 401
  • 种子管理员密码的占位 hash 必须在 M1 末替换为真实 hash
  • 注册接口默认角色为 CANDIDATE，HR 角色仅 ADMIN 可创建（M1 末暴露一个内部脚本即可）

4.4 M3 · 状态机看板（Day 5 – Day 6）⭐ 项目核心亮点

这是项目最核心、最容易出 bug 的模块

状态机 + 审计日志 + 看板拖拽 三者必须强一致。本里程碑 需要单元测试覆盖关键流转分支。

后端：投递 + 流转
M3-BE
实现要点
  • 实体 Application / StageLog + 对应 Mapper
  • ApplicationService：
    • 合法流转图：static final Map<Stage, Set<Stage>> LEGAL_TRANSITIONS

  • 看板接口返回按 stage 分组的 application 数组，控制 N+1 用一次 join
接口

5. 开发环境准备

为什么单独成章

MVP 项目超过一半的时间被环境问题吞掉：JDK 版本不匹配、端口被占、镜像拉不下来、JWT 密钥没生成、 Docker Desktop 没开 WSL2……在 M0 之前花 1–2 小时把环境彻底打通， 能节省后面 3–5 倍的调试时间。本章必须 100% 通过自检后才能进入 M0。

5.2 本地服务方案对比

方案 A：Docker Compose（推荐）
推荐 MVP
优点
  • 一条命令 docker compose up -d 就绪
  • schema.sql 自动初始化到 /docker-entrypoint-initdb.d/
  • 移除 / 重建数据库一条命令，演示数据可反复重置
  • 团队成员环境完全一致
注意
  • Windows 需启用 WSL2 + Docker Desktop 集成
  • 第一次拉镜像可能需配置国内镜像源（见 3.7）
方案 B：本机原生安装
长期开发
优点
  • 启动更快，无 Docker overhead
  • 调试连接、日志、磁盘 IO 更直接
缺点
  • 跨平台差异大（macOS Homebrew / Windows installer / Linux apt）
  • schema.sql 需手动 psql -f 执行
  • 团队成员版本难以严格对齐
建议

MVP 阶段统一使用 方案 A，等正式进入团队协作或部署阶段再考虑混合使用。 Phase 2 技术设计文档已提供 docker-compose.yml 的服务规划。

5.3 凭据与密钥准备

5.3.1 JWT RS256 密钥对（必做）

在仓库外的 ~/.ats/keys/ 或 backend 的 secrets/（已加入 .gitignore）生成密钥对：

# Bash / Git Bash / WSL openssl genrsa -out jwt-private.pem 2048 openssl rsa -in jwt-private.pem -pubout -out jwt-public.pem # 转 base64 写入 .env base64 -w0 jwt-private.pem > jwt-private.b64 base64 -w0 jwt-public.pem > jwt-public.b64 # PowerShell（Windows 等价） openssl genrsa -out jwt-private.pem 2048 openssl rsa -in jwt-private.pem -pubout -out jwt-public.pem [Convert]::ToBase64String([IO.File]::ReadAllBytes("jwt-private.pem")) | Set-Content jwt-private.b64 [Convert]::ToBase64String([IO.File]::ReadAllBytes("jwt-public.pem")) | Set-Content jwt-public.b64

5.3.2 .env 文件准备

  • 后端：ats-backend/.env ← 复制自 .env.example，填入 DB / Redis / JWT base64 值
  • 前端：ats-frontend/.env.local ← 复制自 .env.example，填 VITE_API_BASE_URL
  • 初始管理员账号的 BCrypt hash 生成：
# 用 htpasswd 生成（推荐） htpasswd -bnBC 12 "" "YourAdminPassword" | tr -d ':\n' # 替换 schema.sql 种子数据中的占位 hash
安全红线

.env、*.pem、*.b64、jwt-* 必须全部加入 .gitignore。 切勿在任何 commit / log / 截图中泄露真实密钥与密码。

5.4 代码仓库与 Git 规范

仓库布局

  • MVP 阶段建议 Monorepo： ats/ 根目录下 ats-backend/ + ats-frontend/ + docker-compose.yml
  • 团队规模 > 5 人或前后端发版节奏差异大时再拆 polyrepo
  • 所有文档放 docs/，与 Phase 1/2 产出对齐

.gitignore 必含项

  • .env, .env.local, .env.*.local
  • *.pem, *.b64, secrets/
  • 后端：target/, *.iml, .idea/, logs/, uploads/
  • 前端：node_modules/, dist/, .vite/
  • 系统：.DS_Store, Thumbs.db

分支策略

  • main：保护分支，仅通过 PR 合并
  • feature/m1-auth、feature/m3-pipeline：每个里程碑一条 feature 分支
  • 提交频率：每完成一个 DoD 单元即 commit + push

Commit 规范

  • 格式：[模块] 动词 + 内容，例：[auth] add refresh endpoint
  • 或采用 conventional commits：feat(auth): ... / fix(pipeline): ...

5.7 常见问题与预案

问题表现解决方案
5432 端口被占 Docker 起 pg 失败：port is already allocated 本机已装 PG，停掉服务或在 compose 中映射到 15432:5432
Maven 拉依赖慢 mvn install 卡在中央仓库 ~/.m2/settings.xml 配置阿里云镜像（mirror 节点）
npm install 卡死 依赖下载超时 npm config set registry https://registry.npmmirror.com
Docker 拉镜像超时 pull access denied / 超时 Docker Desktop → Settings → Docker Engine 添加国内镜像 mirror
Windows Docker 无法启动 提示 WSL2 未启用 启用 WSL2 + Hyper-V，wsl --install + 重启
JDK 版本混乱 编译报错 --release 21 is not supported 用 SDKMAN（mac/linux）或 jEnv 管理多版本；或直接 JAVA_HOME 切换
OpenSSL 在 Windows 找不到 'openssl' is not recognized 装 Git for Windows（自带 openssl）或独立安装 winget install openssl
BCrypt hash 含 $ 写入 .env 变量被 shell 解释截断 用单引号包裹，或转义 \$；或写入 application.yml 配置而非 .env

9. AI 辅助开发 SOP

原则一：契约先行

每次让 AI 写一个接口前，先把 DDL 片段 + 接口约定 + 已有代码风格示例 一起喂给它。 不要让它"猜"字段名或返回结构。

9.1 Prompt 模板

## 任务 实现 POST /api/v1/applications 候选人投递接口。 ## 输入约定 { jobId: number, resumeUrl: string, // 已上传后的 URL yearsExp: number, // 0-50 phone: string // 11 位手机号 } ## 输出约定 ApiResponse<ApplicationVO> ## 业务规则 1. 当前用户必须是 CANDIDATE 2. 该 job 必须 status=OPEN，否则抛 BizException(20001) 3. 同一候选人不能重复投递（依赖 UNIQUE(job_id, candidate_id)）， 触发唯一键冲突时抛 BizException(20003) 4. 初始 stage = APPLIED，同事务写入一条 stage_logs(from=null, to=APPLIED) ## 参考代码风格 <贴 JobService.createJob 的实现 30 行> ## 数据库 DDL（相关表） <贴 applications 与 stage_logs 的 CREATE TABLE>

9.2 AI 生成代码的合理粒度

  • ✅ OK：一次生成一个 Controller + Service + Mapper 三件套（同一业务）
  • ✅ OK：一次生成一个 Vue 页面 + 对应的 api 函数
  • ❌ 避免：一次让 AI 写完整模块（如全部认证）—— 难以审阅
  • ❌ 避免：先让 AI 写完 30 个文件再统一调试 —— 累积错误难定位

9.3 多方案时如何选择

  • 选最接近已有代码风格的那一种，不要混用（如 MyBatis-Plus 已在用，就不要再引 JPA）
  • 方案差异显著时让 AI 列出 trade-off，自己定夺，不要直接接受 "AI 推荐"
  • 遇到 AI 给的方案与 project-tech-design.html 冲突时，以设计文档为准

10. 代码自检清单（每次 AI 生成后必做）

后端 7 项

  • 1. 字段名/类型与 DDL 是否对齐？enum 序列化方式（name 而非 ordinal）
  • 2. 接口是否加 @PreAuthorize 或在 SecurityConfig 配规则？
  • 3. 涉及多表写入是否 @Transactional？
  • 4. 参数校验注解（@Valid + 入参 DTO 的 @NotNull/@Size/...）齐全？
  • 5. 有没有用字符串拼 SQL？必须用 #{} 占位符或 MyBatis-Plus Wrapper
  • 6. 状态机相关代码是否调用了统一校验方法，未自定义判断？
  • 7. 日志里有无敏感字段（密码、token、手机号）？

前端 7 项

  • 1. 类型是否从 types/api.d.ts 引入而非现写？
  • 2. 异步操作是否带 loading？错误是否 toast？
  • 3. 表单提交前是否 formRef.validate()？
  • 4. 路由是否在 meta.roles 声明角色？
  • 5. 是否有 hardcode 颜色（应使用 Naive UI theme token）？
  • 6. 接口失败时本地乐观更新是否有回滚？
  • 7. any 是否已消除？枚举是否从 enums.ts 引用？

11. 单任务完成判定（Definition of Done）

  1. 本地能跑通对应场景（含至少一条 happy path + 一条异常 path）
  2. 接口返回结构符合 { code, msg, data }，HTTP 状态码与 code 一致
  3. npm run lint / mvn verify 无新增报错
  4. 关键逻辑（状态机、文件上传校验等）至少 1 个单测覆盖
  5. 提交前 git diff self-review，删除 console.log / 调试用代码
  6. commit message 遵循 [模块] 动词 + 内容（如 [auth] add refresh endpoint）

12. 风险与应对

高风险
  • 状态机非法流转：单测兜底 + 终态保护 + 数据库 CHECK
  • Token refresh 死循环：拦截器内 refresh 失败必须清状态跳 /login
  • 看板并发拖拽：后端基于 from_stage 校验，UI 失败回滚
  • 文件上传越权下载：所有下载接口必须查 application 归属
中风险
  • AI 生成字段名漂移：契约先行 + 自检清单第 1 项
  • 事务边界遗漏：自检清单第 3 项，复盘时再次确认
  • 分页参数命名不一致：约定 page / pageSize，全局统一
  • CORS / Cookie 问题：开发用 vite proxy 规避，部署用 nginx 反代
低风险
  • 标准 CRUD（jobs / departments）— AI 完成度高
  • 样式 / UI 细节 — 在 M5 集中打磨即可
  • 面试评价模块 — 业务规则简单
