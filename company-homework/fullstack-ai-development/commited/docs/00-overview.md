## 1. 项目目标

从 0 到 1 独立完成人力资源系统中「**组织人事模块**」的完整建设，覆盖**产品设计 + 全栈开发 + 交付上线**三个维度，最终交付一个**可实际运行、内网可访问**的系统，并以规范的 Git 提交沉淀工程过程。

考核重点是「**端到端闭环能力**」——不仅是编码，还包括：需求理解深度、产品结构设计、技术方案合理性、把抽象需求落地为可用系统的执行力，以及对用户体验与可扩展性的思考。因此本项目的成败标准不是「写了多少代码」，而是「**能否讲清楚一条完整的业务闭环并让它跑起来**」。

### 北极星指标（自评）

| 维度 | 达标标准 |
| --- | --- |
| 闭环完整度 | 至少 1 条「数据录入 → 流程流转 → 审批 → 留痕 → 统计可视化」的端到端链路全程可演示 |
| 覆盖广度 | 10 个功能域均有产品定义与数据落点；核心域可交互，扩展域至少有页面骨架 |
| 工程规范 | 提交规范、目录规范、Lint 通过、README 可一键拉起 |
| 产品素养 | 信息架构清晰、关键流程有状态机、权限模型自洽、空/错/加载态完整 |
| 可扩展性 | 多租户/组织树、RBAC、审计、统一分页与错误码等横切能力预留 |

## 2. 范围（Scope）

### 2.1 In Scope —— 10 大功能域（全量铺开）

| # | 功能域 | 本期定位 |
| --- | --- | --- |
| 1 | 组织架构管理 | **P0 核心**：主体/部门/岗位/职级/汇报关系 + 架构图可视化 |
| 2 | 员工花名册 | **P0 核心**：全量字段、多维检索、批量导入导出、列表自定义列 |
| 3 | 档案库 | **P1**：电子档案分类存储、权限控制、借阅与归还 |
| 4 | 入离调转管理 | **P0 核心**：入职/离职/调动/转正全流程 + 审批与记录 |
| 5 | 合同管理 | **P1**：签订/续签/变更/到期提醒 + 合同模板 |
| 6 | 任职奖惩 | **P1**：任职资格、晋升、奖惩、考核结果归档 |
| 7 | 员工关怀 | **P2**：生日/周年/节日提醒，关怀记录与统计 |
| 8 | 统计分析 | **P0 核心**：人员结构、入离职趋势、司龄分布等多维看板 |
| 9 | 人事报表 | **P2**：报表模板、自定义生成、导出 Excel/PDF、定时推送 |
| 10 | 人事动态 | **P0 核心**：人员变动时间线 + 操作日志全量留痕 + 变更追溯 |

> 优先级仅用于**排期与降级策略**（见 `06-roadmap.md`），产品与数据设计本期对 10 个域全部给出方案（「全量铺开」）。P0 必须可交互演示；P1 至少打通主链路；P2 给出页面骨架 + 数据结构 + 至少一个可用动作。

### 2.2 Out of Scope（本期不做 / mock）

- 薪酬绩效核算、社保公积金、招聘 ATS、培训 LMS、考勤排班 等其它 HR 子模块。
- 真实SSO 对接、真实短信/邮件网关、对象存储（MinIO/OSS）—— 以 **mock / 占位 / 本地实现** 替代，但**预留接口抽象**。
- 复杂可视化报表设计器、AI 问答（参考工程有，但非本模块重点）。

### 2.3 假设与约束

- 单人独立开发，工期极紧（截止当天）；
- 部署目标为内网一台机器，使用 Docker Compose 一键拉起（DB + Redis + 后端 + 前端）。
- 数据为脱敏/造数，不接入真实员工 PII。

## 4. 技术选型摘要

| 层 | 选型 | 理由 |
| --- | --- | --- |
| 前端框架 | **Nuxt 3**（Vue 3 + TS） | 用户指定；SSR/SEO 友好、目录约定式路由、`server/` 可做 BFF；与 Vue 生态一致 |
| UI/样式 | Tailwind CSS + 设计 token + 组件库 | 与参考工程一致，配 `huashu-design` 设计语言落地 |
| 表格/图表 | TanStack Table + ECharts(vue-echarts) | 花名册大表格 + 统计看板 |
| 状态管理 | Pinia | Vue 官方推荐 |
| 后端框架 | **Go 1.26 + Gin** | 用户指定；参考工程同构，约定可复用 |
| ORM | Bun ORM | 参考工程同款，PG 友好、轻量 |
| 数据库 | **PostgreSQL 16** | 用户指定；树形/JSONB/全文检索能力满足组织人事 |
| 缓存 | Redis 7 | 会话、提醒任务、热点缓存 |
| ID | Snowflake（BIGINT） | 与参考工程一致，避免自增暴露规模 |
| 鉴权 | JWT | 参考工程一致 |
| 部署 | Docker Compose | 一键内网拉起 |
| 设计/演示 | huashu-design（HTML 原型）+ Slidev | 高保真原型、汇报演示 |

> 与参考工程的差异点：前端从 Vite-SPA 改为 **Nuxt 3**（用户指定）。因此前端**不直接复用**参考工程代码，但其组件分层（`components/ui`、`composables`、`stores`、`api`）思路保留；后端**沿用**参考工程的 `internal/<module>`（handler/service/repo/models/router）分层与 migrations 约定。详见 `02-architecture.md`。

## 6. 风险概览（详见 `06`）

| 风险 | 等级 | 应对 |
| --- | --- | --- |
| 工期极紧、范围过大（10 域全铺） | 高 | 严格 P0 优先、扩展域只做骨架；可降级清单前置 |
| Nuxt 与参考工程前端不同构，无现成代码 | 中 | 前端只借鉴分层，组件用 Tailwind 自建；优先用成熟 headless 组件 |
| 审批流状态机复杂度 | 中 | 用通用 `approval_instances` 表 + 简单串行审批，先跑通再增强 |
| 部署/内网联通 | 中 | Docker Compose 一键化，提前验证端口与可达性 |
