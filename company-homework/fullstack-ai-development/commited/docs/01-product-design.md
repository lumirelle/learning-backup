## 1. 产品定位与设计原则

**组织人事管理系统（OrgHR）** 面向集团型企业的 HR / HRBP 与各级管理者，提供「**组织 — 人 — 事 — 档**」一体化的人事运营底座：以**组织架构**为骨架、**员工花名册**为主数据、**入离调转**为状态驱动、**动态/审计**为留痕，向上输出**统计分析与报表**。

设计原则：

1. **以人事档案为单一主数据**：员工是核心实体，所有事件（入离调转、合同、奖惩、关怀）都挂在员工时间线上。
2. **一切变更可追溯**：任何写操作进审计日志，关键业务变更进「人事动态」时间线，可回放员工的全生命周期。
3. **流程驱动状态**：入职、离职、调动、转正、合同续签等都是带审批的状态机，状态变更才驱动主数据更新（而非直接改）。
4. **组织视角与个人视角双线**：既能从组织树俯视，也能从单个员工档案纵览。
5. **可扩展优先**：多组织（租户）、RBAC、字段可配置、列表自定义列等横切能力从一开始预留。

## 2. 用户角色与权限模型（RBAC）

### 2.1 角色

| 角色 code | 名称 | 典型职责 | 数据范围 |
| --- | --- | --- | --- |
| `super_admin` | 超级管理员 | 系统配置、组织/角色/字典维护、全量数据 | 全租户 |
| `hr_admin` | 人事管理员 | 花名册、入离调转、合同、档案、报表 | 本组织及下属 |
| `hrbp` | HRBP | 负责片区/BU 的人事操作与审批 | 授权组织子树 |
| `manager` | 部门经理 | 查看本部门花名册、发起/审批本部门流程 | 本部门子树 |
| `employee` | 普通员工 | 查看本人档案、提交本人相关流程（如离职申请） | 仅本人 |
| `auditor` | 审计员 | 只读全量 + 操作日志/动态 | 全租户只读 |

> 角色为「内置角色 + 数据范围」二维模型。**功能权限**由角色 → 权限点（permission code，如 `employee:export`）映射；**数据权限**由「组织子树 path 前缀 + 本人」规则约束。详见 §2.3。

### 2.2 权限点（示例，完整清单见 `04-api-design.md` 的接口鉴权列）

`org:read/write`、`employee:read/write/export/import`、`archive:read/borrow/admin`、`transfer:apply/approve`、`contract:read/write`、`reward:write`、`care:read`、`stat:read`、`report:read/export`、`audit:read`、`admin:role/dict/config`。

### 2.3 数据权限规则

- **组织子树**：用户绑定一个 `org_path`（如 `/g/sub_a/dept_eng`），可见数据 = `path LIKE '<org_path>%'`。`super_admin`/`auditor` 不受限。
- **本人例外**：`employee` 角色额外可见 `employee.user_id = 当前用户`。
- **字段级**：敏感字段（身份证号、银行卡、薪资带宽、家庭住址）由 `pii` 权限点控制是否脱敏返回（默认掩码）。

## 3. 信息架构（导航）

```

│   ├─ 架构图（可视化树/图）

│   └─ 汇报关系

├─ 人事事务 Affairs

│   └─ 我的审批 / 我发起的
├─ 合同管理 Contracts          合同列表、到期提醒、模板
├─ 档案库 Archives             档案分类、借阅/归还
├─ 任职奖惩 Performance         任职资格、晋升、奖惩、考核归档
├─ 员工关怀 Care               关怀日历、关怀记录与统计
```

## 4. 功能域详细设计

> 每个功能域统一给出：**功能说明 → 关键实体 → 主要页面 → 核心交互/状态机 → 权限点 → 验收要点**。

### 域 1 · 组织架构管理（P0）

- **功能说明**：维护公司主体（集团/子公司/BU）、部门层级、岗位体系、职级体系、汇报关系，并提供架构图可视化。
- **关键实体**：`organizations`（主体树）、`departments`（部门树）、`positions`（岗位）、`job_levels`（职级序列）、`reporting_lines`（汇报关系，亦可由 dept.head + employee.manager 推导）。
- **主要页面**：组织树左侧导航 + 右侧明细；架构图（缩放/展开/导出 PNG）；岗位/职级字典维护页。
- **核心交互**：
  - 组织/部门支持新增、改名、移动（拖拽换父级，更新 `path`）、停用（软删）。
  - 岗位归属部门，含编制数；职级序列（如 P 序列 P3–P10、M 序列 M1–M5）。
  - 架构图按 `path` 渲染，可切换「组织视图 / 部门视图 / 汇报视图」。
- **权限点**：`org:read` / `org:write`。
- **验收要点**：能建一棵 3 层组织树并在架构图正确渲染；移动节点后下属 `path` 级联更新；停用节点不可挂新员工。

### 域 2 · 员工花名册（P0）

- **功能说明**：员工基本信息全量字段管理、多维检索、批量导入导出、列表自定义展示列。
- **关键实体**：`employees`（人事档案主表，含基础/合同/教育/紧急联系人等分组字段）、`employee_field_defs`（可配置字段）、`user_list_prefs`（用户的列表列偏好）。
- **主要页面**：
  - **员工列表**：顶部多维筛选（组织、部门、岗位、职级、在职状态、入职时间段、关键字），表格支持列显隐/排序/冻结、分页、批量操作、导出当前结果、模板导入。
  - **员工档案详情**：头部信息卡 + Tab（基本信息 / 任职履历 / 合同 / 档案 / 奖惩 / 关怀 / 动态时间线）。
- **核心交互**：
  - 批量导入：下载模板 → 上传 Excel → 校验报告（行级错误）→ 确认入库。
  - 导出：导出当前筛选结果为 Excel（受 `pii` 权限脱敏）。
  - 自定义列：用户级保存列配置（`user_list_prefs`）。
- **权限点**：`employee:read/write/import/export`、`pii`。
- **验收要点**：100+ 造数下检索 < 300ms；导入 50 行含 3 行错误能精确报错并部分成功可选；导出文件可打开；列偏好刷新后保留。

### 域 3 · 档案库（P1）

- **功能说明**：电子档案集中管理、分类存储、权限控制、借阅与归还。
- **关键实体**：`archive_categories`（档案分类）、`archive_items`（档案条目，关联 employee + 附件）、`archive_borrows`（借阅单，状态机）。
- **主要页面**：档案分类树 + 档案列表；借阅申请/审批；我的借阅。
- **核心交互（借阅状态机）**：`借阅申请 → 审批中 → 已借出 →（到期提醒）→ 已归还 / 逾期`。
- **权限点**：`archive:read/borrow/admin`、`pii`。
- **验收要点**：上传档案（附件用本地/DB 存储占位）；发起借阅走审批；归还后状态与可借标记复位。

### 域 4 · 入离调转管理（P0 核心）

- **功能说明**：入职、离职、岗位调动、转正全流程，含审批与全程记录。这是系统的**核心闭环**。
- **关键实体**：`hr_processes`（统一人事流程单：type=onboard/offboard/transfer/regularize）、`approval_instances` + `approval_steps`（通用审批）、变更落地到 `employees` 与 `employee_job_history`。
- **主要页面**：四类流程的发起表单、流程详情（含审批轨迹与时间线）、我的审批、我发起的。
- **核心状态机**（通用）：`草稿 → 待审批 →（多级审批）→ 通过 → 已生效 / 驳回 / 撤销`。
  - **入职**：填写拟入职信息 → 审批通过 → 生成员工档案（状态=在职/试用）。
  - **转正**：试用期员工发起 → 审批 → 更新 `employment_status=正式` 并记录转正日期。
  - **调动**：选择目标组织/部门/岗位/职级 → 审批 → 更新员工归属 + 写 `employee_job_history`。
  - **离职**：发起（离职类型/最后工作日）→ 审批 → 到期或即时置 `离职`，保留档案。
- **权限点**：`transfer:apply` / `transfer:approve`（按域复用，命名见 §4 与 `04`）。
- **验收要点**：一条「发起调动 → 二级审批 → 通过 → 员工部门变更 + 履历新增 + 动态时间线新增 + 审计留痕」全链路可演示且数据一致。

### 域 5 · 合同管理（P1）

- **功能说明**：劳动合同签订、续签、变更、到期提醒、合同模板管理。
- **关键实体**：`contracts`（合同，关联 employee，含期限/类型/状态）、`contract_templates`（模板）。
- **主要页面**：合同列表（按到期排序、状态筛选）、合同详情、模板管理、到期提醒看板。
- **核心交互**：签订 → 生效；到期前 N 天进提醒；续签生成新合同并关联上一份；变更留痕。
- **权限点**：`contract:read/write`。
- **验收要点**：能基于模板创建合同；到期 30/15/7 天进工作台提醒；续签链路可追溯。

### 域 6 · 任职奖惩（P1）

- **功能说明**：任职资格评定、晋升记录、奖惩记录、考核结果归档。
- **关键实体**：`qualifications`（任职资格）、`promotions`（晋升，复用流程审批）、`rewards_punishments`（奖惩）、`appraisals`（考核结果归档）。
- **主要页面**：员工档案下的「任职奖惩」Tab + 独立的奖惩/晋升列表。
- **核心交互**：奖惩录入（类型/事由/附件/生效日）；晋升可走审批并联动职级变更；考核结果批量导入归档。
- **权限点**：`reward:write`、`qualification:write`。
- **验收要点**：奖惩记录出现在员工时间线；晋升通过后职级更新且可追溯。

### 域 7 · 员工关怀（P2）

- **功能说明**：生日提醒、入职周年、节日关怀、关怀记录与统计。
- **关键实体**：`care_rules`（关怀规则：生日/周年/节日）、`care_records`（关怀记录）。
- **主要页面**：关怀日历（本月生日/周年名单）、关怀记录、关怀统计。
- **核心交互**：系统按规则生成待关怀名单 → 标记已关怀（方式/备注）→ 计入统计。
- **权限点**：`care:read/write`。
- **验收要点**：当月生日/周年名单正确（按 `birthday`/`hired_at`）；标记关怀后统计更新。

### 域 8 · 统计分析（P0）

- **功能说明**：人员结构分析、入离职趋势、司龄分布、多维度数据看板。
- **关键实体**：以视图/聚合查询为主（`employees`、`hr_processes`、`employee_job_history`），可加物化视图缓存。
- **主要页面**：统计看板（指标卡 + ECharts 图表网格），支持组织/时间维度筛选。
- **核心图表**：
  - 人员结构：按组织/部门/岗位/职级/性别/学历/年龄段分布（饼/柱）。
  - 入离职趋势：按月入职/离职/净增折线。
  - 司龄分布、年龄分布直方图；编制 vs 实有对比。
- **权限点**：`stat:read`。
- **验收要点**：筛选组织子树后所有图表联动刷新；数据与花名册口径一致。

### 域 9 · 人事报表（P2）

- **功能说明**：常用报表模板、自定义报表生成、导出 Excel/PDF、定时推送。
- **关键实体**：`report_templates`（报表模板：字段/筛选/分组定义）、`report_jobs`（生成/推送任务）。
- **主要页面**：报表中心（模板列表）、报表预览、导出、定时推送配置。
- **核心交互**：选模板 → 配筛选 → 预览 → 导出 Excel/PDF；可配置 cron 定时生成并推送（mock 通知）。
- **权限点**：`report:read/export`。
- **验收要点**：至少 3 个内置模板（在职花名册/月度入离职/部门人数）可导出 Excel；定时任务可创建并在日志可见。

### 域 10 · 人事动态（P0）

- **功能说明**：人员变动时间线、操作日志全量记录、变更追溯、审计留痕。
- **关键实体**：`hr_events`（人事动态/业务事件时间线）、`audit_logs`（接口级操作日志，中间件写入）、字段级变更可选 `change_logs`。
- **主要页面**：
  - **人事动态时间线**：全公司/按员工的事件流（入职、调动、转正、奖惩、合同等），可筛选类型/时间/组织。
  - **操作日志**：谁在何时对什么做了什么（method/path/对象/结果），支持检索导出。
- **核心交互**：业务写操作 → service 层发事件 → 落 `hr_events`；所有 HTTP 写请求 → 中间件落 `audit_logs`。
- **权限点**：`audit:read`。
- **验收要点**：任一业务变更都能在动态与审计两条线被检索到；可按员工回放其全生命周期事件。

## 5. 关键业务流程图（核心闭环）

### 5.1 通用审批流程

```
发起人填单 ──提交──▶ [待审批]
                        │  审批人 N 级（串行）
              通过 ┌────┴────┐ 驳回
                   ▼         ▼
              [审批通过]   [已驳回]──▶ 可修改重提
                   │
              业务生效（更新主数据 + 写 hr_events + audit_logs）
                   ▼
               [已生效]
```

### 5.2 入职 → 在职 → 调动 → 转正 → 离职（员工生命周期）

```
拟入职 ─审批─▶ 在职(试用) ─转正审批─▶ 在职(正式)
                  │                      │
              调动审批（部门/岗位/职级变更，写履历）
                  │                      │
                  └──────── 离职审批 ─────┘──▶ 离职（档案留存）
```

每个状态迁移都会：① 更新 `employees` 主数据；② 追加 `employee_job_history`；③ 生成 `hr_events` 时间线；④ 中间件写 `audit_logs`。

## 6. 全局体验规范

- **三态齐全**：每个列表/详情都要有 加载（skeleton）/ 空（EmptyState，含引导动作）/ 错（可重试）三态。
- **危险操作二次确认**：删除、离职生效、批量操作需确认弹窗。
- **乐观反馈 + Toast**：写操作即时 Toast，失败可回滚提示。
- **可达性与一致性**：统一组件（Button/Card/Modal/Drawer/Table/Pagination/Avatar/Badge），统一日期/数字格式与中文文案。
- **权限感知 UI**：无权限的操作按钮隐藏或禁用并解释原因。
