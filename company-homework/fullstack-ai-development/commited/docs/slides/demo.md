## 0 · 准备（起服务）

```bash
# ① 依赖容器（PostgreSQL:5433 / Redis:6380）
mise run infra

# ② 后端 API（端口 8090；首启自动迁移+种子：3 组织/6 部门/30 员工）
APP_PORT=8090 mise run //server:api

# ③ 前端（自动把 /api/v1 反代到 8090；3000 被占会顺延端口）
mise run //web:dev
```

打开终端里 `Local:` 提示的地址（如 `http://localhost:3000`）。

<div class="text-sm opacity-60 mt-4">
内网部署时由 nginx/Caddy 反代，对外给一个 URL 即可。
</div>

## 1 · 登录

| 操作 | 预期 |
| --- | --- |
| 打开站点 | 未登录自动跳转 `/login` |
| 点「超级管理员」按钮（或输 `super_admin` / `ChangeMe@123`）| 输入框自动填充 |
| 点「登录」 | 进入**工作台**，右上显示用户名 + 角色徽标 |

> 验证点：鉴权生效（无 token 进不去）；token 存 cookie，刷新仍在登录态。

## 2 · 工作台（Dashboard）

看四类信息是否都渲染：

- **指标卡**：在职人数 / 试用期 / 本月入职 / 待审批
- **我的待办审批**（当前为 super_admin，可能为空）
- **合同到期提醒（30 天内）**：含「剩 N 天」徽标
- **最近人事动态**：入职/离职/调动 事件流 + 类型徽标

> 这页的数据分别来自 `/analytics/overview`、`/approvals/todo`、`/contracts/reminders`、`/timeline`。

## 4 · 发起调动（核心闭环 · 起点）

在员工档案点「**发起人事流程**」，或左侧「人事事务」→「**+ 发起流程**」：

| 字段 | 填写 |
| --- | --- |
| 流程类型 | **岗位调动** |
| 员工 | 任选一名在职员工 |
| 目标部门 | 如「产品部」 |
| 目标职级 | 如 `P7` |
| 事由 | 「业务调整」 |
| **审批人** | 选「**研发经理（mgr01）**」|

点「提交审批」→ 跳转流程详情，状态 **待审批**，审批轨迹第 1 步「待审批」。

---

## 5 · 审批（切换到审批人账号）

1. 右上「**退出**」→ 用 **mgr01 / ChangeMe@123** 登录
2. 「人事事务」→「**我的待办**」（或工作台「我的待办审批」）应出现该流程
3. 点进详情 → 底部「**我的审批**」→ 填意见 → 点「**通过**」
4. 预期：状态变 **已生效**，审批轨迹第 1 步「已通过」

<div class="text-sm opacity-70 mt-3">
🔒 越权验证（可选）：用 <b>emp01</b> 登录，看不到该待办；即使打开详情也<b>没有审批按钮</b>（返回 1202/无权限）。
</div>

---

## 6 · 验证闭环一致性（关键）

一次审批通过，应在**五处**同时体现：

| 位置 | 预期 |
| --- | --- |
| 员工档案 | 部门/职级**已变更** |
| 任职履历 | 新增一条 transfer（from→to 快照）|
| 人事动态时间线 | 多一条「岗位调动至 …」|
| 操作日志 | 有 `POST …/approve` 状态 200 |
| 统计看板 | 人员结构 / 编制实有**已联动** |

> 这正是 requirements 强调的「**端到端闭环**」：录入 → 流程 → 审批 → 留痕 → 可视化。

## 8 · 人事动态 / 操作日志

- 「人事动态」→「**变动时间线**」：全公司业务事件流（语义留痕）
- 切「**操作日志**」：谁在何时调了哪个接口、状态码（技术留痕 / 审计）

> 业务留痕（hr_events）与技术留痕（audit_logs）双线互补，可追溯、可审计。
